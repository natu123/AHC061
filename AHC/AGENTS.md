# AHC - AGENTS Instructions

## Scope
- このファイルは `AHC/` 配下の共通ルールを定義する
- `../AGENTS.md`（AtCoder共通）を継承する
- 各コンテスト配下（例: `AHC061/AGENTS.md`）は、この共通ルールを継承しつつ固有ルールで上書きできる

## Source Of Truth
- AtCoder共通の運用方針は `../AGENTS.md` を正本とする
- AHC共通の運用方針はこのファイルを正本（single source of truth）とする
- 各コンテスト固有の仕様・手順は各ディレクトリの `AGENTS.md` を正本とする
- ルール変更はまず該当スコープの `AGENTS.md` を更新し、必要な差分のみ他ファイルに反映する

## Communication
- 一人称は「私」を使用する
- 返答は常に敬語（です・ます調）で統一し、断定的な常体（だ・である調）は使わない
- ユーザー向けの返答は全て日本語（英語はコード・ファイルパス等の技術的識別子のみ）
- 受動的な「問い待ち」ではなく、自律的に提案・計画を行うエージェント型の対話
- タスクやフェーズの節目に進捗を要約（Recap）する
- 「終了の提案」や「時刻の通知」は行わず、ユーザー主導の区切りを尊重する
- 機能的限界について嘘やできない約束はせず、事実ベースで対話する

## Smart-Commit Workflow
キリの良いタイミングでコミットを自然に提案する。
**原則: 1変更 = 1コミット**

### コミット前チェックリスト（再発防止）
- コミット前に `git diff --name-only` と `git diff` を確認し、変更を `Feat / Fix / Refactor / Perf / Style / Docs / Chore` 単位で分割する
- 1コミットに複数目的が混在している場合は分割する（例: 探索ロジック変更とログ整形を同居させない）
- 同一ファイル内に複数目的の変更がある場合は、ファイル単位ではなく差分（hunk）単位で分割する
- 差分分割には `git add -p <file>` を使用し、1コミットに含める意図したhunkのみをステージする
- コミット前に目的ラベル（例: `temperature-tuning` / `beam-width-cap` / `io-fastpath`）を列挙し、ラベルごとにコミットを分離する
- 各コミットで「対象」と「目的」が1文で言い切れない場合は、コミット粒度が大きすぎると判断して分割する
- `git add` はコミット単位で実施し、`git add .` の一括ステージは原則禁止（意図しない混在防止）
- push前に「このコミットは1変更か？」を自己確認し、YESの場合のみ `git push` する
- `git commit` 実行前に `git diff --cached --name-only` と `git diff --cached` を確認し、対象ラベル以外が含まれていないことを確認する
- push直前に `git log --oneline -n <分割予定数>` を確認し、事前計画したラベル順で並んでいなければ push しない

### 命名規則（履歴可読性の向上）
- 形式: `<Type>: <対象>を<目的>（必要なら手段）`
- 例1: `Perf: 遷移評価を差分更新化して1stepあたり計算量を削減`
- 例2: `Feat: 焼きなまし初期解に貪欲挿入を追加`
- 例3: `Docs: seed運用ルールをAGENTSに明文化`
- 「調整」「改善」など抽象語だけの件名を禁止する

### コミット本文ルール（箇条書き必須）
- `git commit -m "<件名>" -m "<本文>"` を使用し、本文を必ず付与する
- 本文テンプレート:
  - `- 背景: なぜこの変更が必要か`
  - `- 変更: 何をどう変えたか（主要ファイル）`
  - `- 影響: スコア/実行時間/副作用/確認観点`

## AHC Domain
- コンテスト問題に対するヒューリスティック最適化を対象とする
- 目的は「制約を満たす出力」かつ「スコア最大化（または最小化）」とする
- 単発のACではなく、seed群で安定して高スコアを出す実装を重視する

## LLM Compliance
- AtCoderのLLM利用ルールに従い、APIやスクリプト等による生成AIへの多数出力の自動生成依頼は行わない
- 生成AIの利用は人手主導の対話・検討に限定し、候補の大量生成と自動選別を組み合わせない
- 参考: `https://info.atcoder.jp/entry/ahc-llm-rules-ja`

## Engineering Protocol
- **⚠️ Score-Driven Fix（最重要）**: 変更の目的を必ず「どの指標を何点改善したいか」で定義する
- **Reproducibility First**: seed・実行コマンド・時間制限・コミットIDをセットで記録し、再現できない改善は採用しない
- **Root Cause Analysis**: スコア低下時は現象（結果）と原因（遷移設計/温度設計/初期解品質）を分離して分析する
- **A/B Validation**: 改善主張は同一seed集合で比較し、平均・中央値・最悪ケースを併記する
- **Small Batch Experiments**: まず少数seedで探索し、有望案のみ全seed評価に進む
- **Time Budget Guard**: 本番制限時間を常に意識し、ローカルで上限時間を守るガードを入れる
- **Constraint Safety**: 速度改善より制約違反検出を優先する。違反時は即座にフォールバック可能にする
- **Diff Evaluation**: 近傍遷移は可能な限り差分評価を使い、フル再計算を減らす
- **Deterministic Mode**: 乱数seed固定モードを用意し、デバッグ時は必ず固定で検証する
- **Log Hygiene**: 提出コードには過剰ログを残さず、検証用ログは切替可能にする
- **User-Led Final Check**: 最終提出判断はユーザー主導。AIは比較結果とリスクを明示する
- **Meta Improvement**: 失敗や運用漏れは retro して、同日中に手順・Docs・上位ルールへ反映する
- **Plateau Pivot**: 直近 `8` 試行採用 `0` 件、または `10` 試行平均改善率 `+0.3%` 未満なら、微調整を打ち切り `Exploit 5% / Explore 95%` の抜本探索へ移行する
- **Portfolio Exploration**: 抜本探索時は最低 `2` 系統、推奨 `4` 系統以上の異種アプローチを並列で評価する
- **Adaptive Allocation**: `EGR = P(hit) * Gain / Cost` に基づき、`10` 試行ごとに Exploit/Explore 配分を更新する

## Conventions
- まず既存ファイルを編集し、新規ファイル追加は必要最小限にする
- コメントは「意図・数式・計算量」の説明に限定し、冗長な実況コメントは避ける
- ファイル/変数命名に `tmp`, `new2`, `final_final` のような曖昧語を使わない
- 乱数・定数パラメータは1箇所に集約して実験しやすくする
- PowerShell実行を前提とし、コマンド連結は `;` を使う

## Experiment Management
- 比較は最低でも「baseline」と「candidate」の2系統で行う
- 記録項目: 実行日時、commit、コマンド、seed範囲、平均スコア、中央値、最小スコア、実行時間
- 採用基準: 平均改善だけでなく、最悪ケース悪化と時間超過リスクを許容範囲内に収める
- 大きな変更（探索方式変更など）は段階導入し、1ステップずつ効果検証する
- 試行IDは `T-xxx (Xyy-zz) S/F-kk` を推奨形式とし、全体通番・戦略ファミリ通番・採否通番を併記して管理する

## Solver Lineage
- ソルバ系統IDは `x01, x02, ...` の作成順で採番し、既存IDの意味は変更しない
- `champion` は「現在採用中の系統」を示すラベルとしてのみ扱い、系統IDやファイル名の意味を上書きしない
- 提出物は採用時点の系統IDを含む成果物名（例: `submission_x10.rs`）で生成して追跡可能性を確保する

## Docs Governance
- Docsは「目的別に分離」し、1ファイル1責務を原則とする
- ガイドと検討ログと厳密仕様を同一ファイルに混在させない
- 運用変更は影響スコープのファイルだけに反映し、無関係なmdへの横並び更新は行わない

### Docs分類（共通）
- `*_Codex_Guide.md`: 実行手順・運用ルール・依頼テンプレート
- `*_Game_Rules_Strict.md`: 実装準拠の厳密仕様（状態遷移・判定順序・例外）
- `*_Initial_Study_YYYY-MM-DD.md`: 初期検討プロセスと比較結果
- `*_Experiment_Log_YYYY-MM.md`: 継続検証ログ（月単位またはフェーズ単位）
- `solver_specs_built/`: 作成済みソルバ（xNN）の戦略詳細
- `solver_specs_planned/`: 計画中ソルバ（xNN）の戦略詳細

### ソルバ詳細Docs運用（共通）
- `docs/solver_specs_built` と `docs/solver_specs_planned` を標準フォルダ名として使用し、日本語ディレクトリ名は使用しない
- 作成済み/計画中でファイルを分離し、`xNN`ごとに1ファイルで戦略詳細を管理する
- 新規ソルバ作成時は、実装前に `solver_specs_planned`、実装後に `solver_specs_built` へ追記または移管する

### 命名規則（共通）
- 形式: `<ContestID>_<DocType>[_YYYY-MM-DD].md`
- 日付付きファイルは不変ログとして追記中心で運用する
- ガイド/厳密仕様は固定ファイル名で更新し、履歴はGitで追跡する

### 更新フロー（共通）
1. 仕様変更: 先に `*_Game_Rules_Strict.md` を更新
2. 運用変更: `*_Codex_Guide.md` を更新
3. 検討/比較: 日付付き `*_Initial_Study_*` または `*_Experiment_Log_*` に記録
4. 参照追加: 各コンテストの `AGENTS.md` `Project Documentation` を更新

### 最低記録項目（検討ログ）
- 実行日時
- 対象commit
- 実行コマンド
- seed範囲
- 平均/中央値/最小スコア
- 実行時間
- 採用/不採用判断と理由
- 不採用時は再発防止タグ（例: `tail-risk-worsen`, `no-op`, `beam-over`）を付与する
